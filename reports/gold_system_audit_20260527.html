🚨 P0 关键问题 (必须修复)

安全 P0-1: API Key 硬编码在源码中
gold_config.py 中 siliconflow_api_key 和 openrouter_api_key 明文硬编码。FRED API Key 也以默认值写入代码。任何能访问代码仓库的人都能获取这些密钥。
🔧 修复方案: 所有 API Key 改为仅从环境变量读取，无默认值。添加 .env 文件支持 (python-dotenv)，.env 加入 .gitignore。
架构 P0-2: 报告生成器 v5/v6 代码重叠 58%
v5 (3,250行) 和 v6 (3,399行) 之间存在 1,306 行重复代码。两个文件合计 6,649 行，实际有效代码仅约 4,243 行。维护时改一处漏一处，Bug 修复不同步。
🔧 修复方案: 将 v5/v6 公共逻辑抽取为 GoldReportBase 基类，v5/v6 继承并实现差异化部分。预计可削减 1,500+ 行重复代码。
架构 P0-3: 模块级全局单例导致启动耦合
data_service.py 末尾 DATA_SERVICE = DataService() 在模块导入时即触发所有 9 个 Provider 初始化。任何 import 都会导致网络连接检测、API 调用等副作用，且无法延迟加载或注入依赖。
🔧 修复方案: 改为懒加载模式 def get_data_service(): global DATA_SERVICE; if DATA_SERVICE is None: DATA_SERVICE = DataService(); return DATA_SERVICE
质量 P0-4: 53 处裸 except + pass 吞异常
全项目 53 处 except: pass 或 except Exception: pass，分布在 quantitative_strategy(8处)、providers(29处)、report_generator(14处)。异常被静默吞掉，导致调试困难，数据缺失时无感知。
🔧 修复方案: 全部改为 except Exception as e: logger.warning(f"xxx失败: {e}")，至少记录日志。
质量 P0-5: 零单元测试覆盖
17 个测试文件全部是集成/手动调试脚本 (test_debug_crash, test_suanli_api 等)，没有任何 pytest 单元测试。核心模块 (factors, strategies, risk_manager) 的计算逻辑完全无自动化验证。
🔧 修复方案: 建立 tests/ 目录，为核心计算模块编写 pytest 单元测试。优先级: factors > strategies > risk_manager > backtester。

⚠️ P1 重要问题 (应尽快修复)

性能 P1-1: 数据获取全部串行，无并发
DataService.get_all_data() 顺序执行 5 步 (价格→宏观→情绪→相关性→质量)，每步可能超时。在9个Provider中只有第一个成功的被使用，其余全部白跑。
🔧 修复方案: 使用 concurrent.futures.ThreadPoolExecutor 并行获取数据。同类型 Provider 竞争式获取（谁先返回用谁），不同类型并行获取。
质量 P1-2: 3处 fillna(method=...) 弃用写法
factors.py L56、risk_manager.py L87、data_fusion.py L71 使用 fillna(method='ffill')，pandas 2.2+ 已弃用，未来版本将移除。
🔧 修复方案: 替换为 .ffill() / .bfill()。
架构 P1-3: 无依赖管理 (缺 requirements.txt)
项目既无 requirements.txt 也无 pyproject.toml。新环境部署时无法一键安装依赖，版本兼容性无保证。
🔧 修复方案: 生成 requirements.txt (至少包含 akshare, numpy, pandas, requests)，长期迁移到 pyproject.toml。
策略 P1-4: 策略信号阈值硬编码
strategies.py 中买入信号阈值 > 1%、卖出信号 < -0.5% 硬编码，未根据市场波动率自适应调整。在低波动环境下信号过于敏感，高波动环境下又太迟钝。
🔧 修复方案: 改为基于 ATR 或历史波动率的动态阈值：threshold = base + k * realized_vol
架构 P1-5: Provider 接口不统一
9个Provider中仅 akshare 和 china_http 实现了 get_gold_price + get_macro_data + get_market_sentiment 完整三接口。goldapi 只实现 get_gold_price，openbb 缺 get_asset_correlation，tencent 缺 get_macro_data。基类 DataProvider 的抽象方法定义与实际实现脱节。
🔧 修复方案: 在 DataProvider 基类中明确声明 capability 属性，DataService 根据 capability 路由调用，不再假设每个 Provider 都有全部接口。
质量 P1-6: 根目录 33 个脚本文件混乱
根目录有 33 个 .py 文件（含 _diagnose_central_bank.py、fix_gold_report.py、test_debug_crash*.py 等），无分类无组织。临时脚本、调试代码、一次性修复脚本与正式代码混在一起。
🔧 修复方案: 建立 scripts/ 目录存放工具脚本，tests/ 目录存放测试，根目录仅保留 run_gold_report.py 等入口文件。
